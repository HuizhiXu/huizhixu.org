前段时间，Hermes Agent 项目在社区兴起，我也去研究了一番。在深入了解后，我忽然意识到一个事情：**工程层面，大家都在把 Agent 系统做得越来越复杂、越来越重。但对于个人使用者而言，最重要的资产其实是我们自己的知识库。**

我之前一直用 Notion，但是它也有缺点：启动慢；工作记录和个人生活记录混在一起，需要查询时很不方便。Obsidian 是与 Notion 齐名的另一款笔记软件，过去你可能随便选一个就行。**但进入 AI 时代，Obsidian 多了 Notion 不具备的两个优势：1. 本地存储；2. 开源生态**。

另外Obsidian使用 Markdown 格式，天然适合 AI 阅读。于是我萌生了一个想法：把工作中涉及的技术、思考、职场沟通表达、读书笔记等，全部放到到一个专属知识库中。这些内容是我工作中随时要用的，即使当着同事面查找，也不会暴露不想让人看到的私人信息。

所以过去的几个礼拜，我一直在搭建我的Obsidian 的知识库。

它的安装和使用都非常简单，且支持多平台。还能像编程 IDE 一样自由选择字体、配色和界面风格。除了安装之外，**我还用家里的NAS倒腾了多平台同步，官方的一个月5美金的云同步也是省下了。**

我在用Notion时发现，自己想记录一个东西，却不知道该放哪个文件夹——它有时属于工作，有时属于生活，或者生活中的某个细分类别。

我试过用“技术/子技术”的方式来组织，但每个项目涉及的技术、实验、结果和进度管理，本应属于不同领域，却被我混在一起。**例如，随着AI领域不断扩大，我接触的AI产品也越来越多，有时要记录产品本身，有时记录使用心得，有时记录技术路线，导致我的AI相关资料越来越多，却分散在各个地方，导致很多内容没能有效沉淀。**

这次我借着 Obsidian 好好整理了一套适合自己的知识库结构，这篇文章主要想分享这套思路。它的核心清晰明了：每个内容都有明确的存放位置，再也不用像以前那样，想写点什么时找不到地方。

## 开始的设计理念

最初，我打算按**身份**来分类。

我的日常工作围绕 AI 算法，有时要探索 Agent 和 AI 产品，既看论文又写代码，偶尔还写文章、做读书笔记。从这些活动里可以提炼出四类身份：**开发者、研究者、读者、写作者**。于是一个直观的树状结构浮现出来：

```plain text
├── 开发者 
	├── 技术文档
	├── 开发的项目 
	└── 学习笔记与总结 
├── 研究者 
	├── 论文笔记与文献管理 
	├── 研究思路整理
	└── 论文复现实验记录
├── 读者 
	├── 读书笔记 
	└── 个人知识库建设 
├── 写作者 
	├── 创意写作与故事构建 
	├── 公众号内容规划 
	└── 小红书写作 
```

但这个看似清晰的树状结构，在实际使用中很快就暴露了问题：

> 一篇论文笔记，它既是“研究”的一部分，也可能包含“代码实现”，还可能成为“写作”的素材。那它到底该放哪儿？

答案又开始含糊不清。含糊不清就会导致混乱。《认知觉醒》里面说模糊就是认知最大的敌人。

于是我调研了一下大家是如何构建笔记系统的，进而发现了 PARA 这种方式非常适合我的方式。**PARA 方法（Project, Area, Resource, Archive），它的分类逻辑从按身份转向按工作流 / 项目。**

## 设计理念

PARA 包含四个顶级类别：

- **Projects (项目)**：有明确截止日期和具体成果的任务。
- **Areas (领域)**：需要持续负责、没有完成一说的工作或生活板块。
- **Resources (资源)**：一个主题知识库，像私人的 Wikipedia，备查即可。
- **Archive (归档)**：已完成的 Projects、过时的 Areas 或不再需要的 Resources。
以下是我的 Obsidian 根目录结构：

```plain text
Obsidian 根目录
├── 00_热区 (Current Focus)          // 当前工作的聚焦看板
│   ├── Dashboard.md                // 记录当前探索方向、关键论文、实验链接
│   ├── CS336_2026Spring_MOC.md     // 课程进度
│   └── Obsidian学习.md             // 待解决的问题
├── 01_工作流核心 (Workflow)         // 日常高频使用区
│   ├── Projects/                   // 具体、有时限的任务
│   ├── Areas/                      // 持续关注的长期责任
│   └── Resources/                  // 主题知识库
├── 02_输出与创作 (Outputs)          // “作品”区
│   ├── Drafts/                     // 文章草稿、博客、技术报告
│   └── Snippets/                   // 可复用代码与实验记录
├── 03_个人系统 (Personal)           // 与工作无关的个人成长与兴趣
│   ├── Reviews/                    // 读书笔记、读后感
│   ├── Methods/                    // 沟通、时间管理、理财等原则
│   └── History & Interests/        // 纯粹的兴趣内容
└── 04_存档与收件箱 (Archive & Inbox)
    ├── Inbox/                      // 快速捕获，定期整理
    └── Archive/                    // 已完成项目、过时笔记
```

## 01_工作流核心 (Workflow)

这是我每天进出最频繁的目录。

### `Projects/`（项目）

- **定义**：有明确起止时间、可交付成果的任务。
- 每个项目 = 一个文件夹 + 一个同名的入口文档，这样在Dashboard上链接比较好。
- **例子**：
```plain text
   01_工作流核心/Projects/
   ├── Hermes Agent深度体验/ # 文件夹
   │   ├── Hermes Agent深度体验.md# 入口文档（与文件夹同名）
   │   ├── 安装使用笔记.md
   │   ├── 原理分析.md
   │   └── 实验记录.md
```

- **如何用**：每个项目一个文件夹，里面放该项目相关的所有笔记（会议记录、代码片段、阅读清单）。项目结束后**整个文件夹移到 `04_存档/归档`**。
### `Areas/`（领域）

- **定义**：需要持续负责、没有完成一说的工作领域。
- **例子**：
- `LLM论文追踪`（每周读新论文，维护一个MOC）
- `游戏里面NPC的构建`
- `生图生视频`（长期关注）
- **如何用**：每个领域一个文件夹，里面放**该领域的核心MOC（内容地图）**，比如 `LLM论文追踪/MOC_2026.md`，然后通过链接指向具体笔记。领域中的笔记可以很零散，但MOC把它们串起来。
- **注意**
- 项目总是属于领域
- 注意
- **坚持跳舞是一个项目，而个人身体健康是一个领域**
- **学完 CS336 课程是一个项目，而大模型是一个领域**
### `Resources/`（资源）

- **定义**：主题知识库，存知识的地方，像一个私人的Wikipedia。
- **例子**：
- `Transformer架构`（下含 `Attention机制详解.md`、`位置编码对比.md`）
- `提示工程`（下含 `CoT技巧.md`、`few-shot示例库.md`）
- `工具`（下含`NAS使用.md`，`Obsidian使用.md`)
- **如何用**：按主题建子文件夹，每条笔记尽量原子化（一个概念一个文件）。
- **不要在这里放待办或项目相关的东西**。
## 02_输出与创作 (Outputs)

这里放我的输出作品（包括未完成的作品）。与 `Projects` 不同的是，`Projects` 存放过程笔记，而 `Outputs` 存放最终或半最终产物。

### `Drafts/`（文章草稿）

- 所有写作任务的大纲、初稿、修改稿。
- **例子**：`Langgraph实用指南_草稿.md`，内部可以链接到 `Resources/Agent` 和 `Areas/LLM论文追踪` 中的笔记。
## 03_个人系统 (Personal)

与工作无直接关系的笔记。

### `Reviews/`（读后感）

- 读书笔记和读后感
- **例子**：`李诞工作手册.md`、`维特根斯坦传.md`。
### `Methods/`（方法论）

- 沟通技巧、时间管理、理财原则、决策框架。
- **例子**：`非暴力沟通四步法.md`、`GTD流程.md`、`资产配置原则.md`。
### `History & Interests/`（历史与兴趣）

- 出于好奇和兴趣而研究的知识。名字叫History是提醒自己尽量多了解一些历史。
- **例子**：`河西走廊历史.md`
## 04_存档与收件箱 (Archive & Inbox)

### `Inbox/`（收件箱）

- **`Inbox/` (收件箱)**：快速捕获任何想法、网页剪藏、临时笔记。每月定时清空，分类整理。
### `Archive/`（归档）

- 已完成的项目、过时的领域笔记、不再维护的资源。不要删除，只是移到这里。Obsidian 的搜索仍然能搜到，但不会干扰当前工作流。

## 为什么这套结构对我有用？

一开始，我很难分清 Projects、Areas 和 Resources 的区别，每当要开始一个新项目，都不知道该放哪里。经过几周实践，我总结出一个**最重要的判断准则**：

> **如果这个文件夹里的内容最终要产出一个东西（报告、代码、文章），那就放到 Project 里面。** 没有明确产出，则放入 Resources 或 Areas。

例如，在 `Areas/LLM论文追踪` 下有一个“读 VoxCPM2 论文并完成声音克隆”的子任务。这个子任务有明确目标（完成声音克隆），因此我将它单独作为一个 **Project**，放在 `Projects/VoxCPM2声音克隆` 里。项目完成后存档，而 Areas 里只保留长期维护的内容（比如“VoxCPM2 最佳实践笔记”）。

**刚开始处理一两个项目时可能会有点纠结，但到第三个项目我就得心应手了。**

于是这几周，我放了目前需要的技术笔记和工作项目，所以 `Projects` 和 `Resources` 是改动最多的部分。其他模块还在搭建中，比如：

### 读书笔记：

1. 读书笔记放在03_个人系统/02Histories & Interests。
1. 读后感放在03_个人系统/03Reviews 读后感观后感。
1. 输出的放在02_输出与创作_Outputs/Drafts。
1. 书籍提炼的方法论就放在Methods里面。
### 公众号文章：

1. 公开发表的文章放在Drafts里面。
1. 发表之后归档在04存档与收件箱/01Archive/Published/Book_G。
1. 公众号中用到的**临时配图**，放在每篇文章同级目录下的 `attachments/` 文件夹（Obsidian 可配置附件路径）。
1. 如果某个图片会被多篇文章复用（如头像、logo），放在 `Resources/媒体素材/` 下。
1. 当需要引用 Resource 中的**具体数据或结论**时，用 **Obsidian 的块引用**（`^blockId`）或者简单的链接。**不要全文嵌入**。
**这两个场景我还在持续实践，肯定会多次调整。但我相信，一旦形成稳定的工作流，它就能无限复用。**
